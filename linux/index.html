00 · foundation

What is the Kernel? And Why Does it Exist?

Think of your computer as a city. Your apps (browser, terminal, VS Code) are like businesses. The kernel is the city government — it owns all the roads, electricity, water. No business gets direct access to infrastructure. They file a request, the government decides, then handles it.

🏛️
What the kernel owns

The kernel has exclusive control over CPU, RAM, disk, and all hardware. User programs are guests — they can only touch hardware by politely asking the kernel through system calls.

🎭
Why this separation?

Imagine if any app could directly write to your disk or grab all the RAM. Chaos. The kernel is a trusted middleman that enforces rules: process A can't read process B's memory. Chrome can't accidentally crash your disk driver.

⚙️
What the kernel does

It manages 5 core things: who gets CPU time (scheduler), virtual memory, file system access, hardware communication, and network I/O. Everything else is userspace.

💤
When does the kernel run?

The kernel is event-driven. It doesn't run constantly — it sleeps. It wakes up when: a program makes a syscall, a hardware interrupt fires, or a timer tick happens. Otherwise, it's out of the way.

💡
The Protection Rings — CPU-level enforcement
The x86 CPU has 4 privilege rings (0–3). Linux uses Ring 0 (kernel mode) and Ring 3 (user mode). In Ring 0, code can execute any instruction and access any memory. Ring 3 code gets a page fault or exception if it tries. This is enforced in hardware — it's not just a software convention.

01 · boot sequence

From Power Button to Kernel Alive

The moment you press power, your machine goes through a carefully choreographed sequence. Nothing in the kernel is running yet — the CPU wakes up cold and has to be handed the wheel, step by step.

🔌
Step 1 — BIOS / UEFI

The CPU wakes up at a fixed address (0xFFFF0 in ROM). It runs POST (Power-On Self Test) — checks if RAM is there, if the CPU cores respond. This is the firmware, not Linux. On modern machines, UEFI replaces legacy BIOS with a full mini-OS that can read FAT32 partitions.

💿
Step 2 — MBR / GPT

BIOS reads the first 512 bytes of your boot disk — the Master Boot Record. It contains a tiny bootstrap program. GPT (modern disks) stores a partition table that UEFI reads to find the EFI System Partition. This stage's only job: load the real bootloader.

🥾
Step 3 — GRUB2

GRUB (Grand Unified Bootloader) is the menu you sometimes see on boot. It knows how to read ext4/BTRFS/NTFS, finds vmlinuz (the compressed kernel) and initrd (initial RAM disk), loads them into RAM, and jumps into the kernel's entry point.

🌱
Step 4 — initrd & start_kernel()

initrd is a tiny temporary filesystem in RAM with just enough to mount the real root filesystem. Once mounted, start_kernel() runs — setting up the memory allocator, interrupts, scheduler, and finally spawning PID 1 (systemd).

🔍
What is vmlinuz exactly?
vmlinuz = "Virtual Memory LINUx, gZipped". It's the Linux kernel compressed with gzip/zstd. GRUB loads it into RAM, and the first thing the kernel does is decompress itself. The z at the end literally means zipped. You can see it at /boot/vmlinuz-6.8.0-generic on Ubuntu.
02 · process management

The Scheduler — Who Gets the CPU?

You have 200+ processes running right now but only 4–8 CPU cores. How does everyone get a turn? That's the scheduler's job. Linux uses CFS — the Completely Fair Scheduler. The idea: every process deserves its fair share of CPU time, no more, no less.

🏃
Process States

Every process is in one state at a time:
🟢 Running — actively on a CPU core
🔵 Waiting — in queue, ready to run
🔴 Blocked — waiting for I/O (sleeping)
⚪ Stopped — paused (Ctrl+Z)
🟡 Zombie — finished but parent hasn't read exit status yet

🌳
CFS and vruntime

Each process has a virtual runtime (vruntime) counter. When a process uses CPU, its vruntime ticks up. CFS always picks the process with the lowest vruntime — the one that's had the least CPU recently. This is stored in a red-black tree (self-balancing, O(log n) lookup).

🔄
Context Switch

When the scheduler swaps processes, it saves the current process's CPU state (registers, stack pointer, instruction pointer) into its task_struct, then loads the next process's state. This is called a context switch. Takes ~1–10 microseconds.

📋
task_struct

Every process is represented in the kernel by a task_struct — a C struct with 600+ fields: PID, parent PID, memory maps, open files, signal handlers, scheduling info, and more. Run cat /proc/$$ in a terminal to peek at your shell's info.

// How CFS picks the next process — kernel/sched/fair.c (simplified)
struct task_struct *pick_next_task_fair(struct rq *rq) {
  struct sched_entity *se;
  // Grab the leftmost node from the red-black tree
  // Leftmost = smallest vruntime = "most deserving" process
  se = __pick_next_entity(cfs_rq);
  if (!se) return NULL; // no runnable tasks
  set_next_entity(cfs_rq, se); // mark it running
  return task_of(se); // return the task_struct
}
03 · memory management

Virtual Memory — The Grand Illusion

Every process on your machine thinks it has the entire 64-bit address space to itself — up to 128 TiB. Obviously that's not physically true. The kernel (with CPU hardware help) creates this illusion using virtual memory and page tables.

🗺️
Virtual vs Physical Addresses

When your C program does malloc(), it gets a virtual address. The CPU's MMU (Memory Management Unit) silently translates it to a physical RAM address using a page table. Each process has its own page table — that's why process A can't read process B's memory even at the same address.

📄
Pages — 4KB at a time

Memory is divided into 4KB chunks called pages. The kernel doesn't track individual bytes — it tracks pages. When you malloc 1 byte, you get a whole 4KB page. Page tables map virtual page numbers to physical page frames. On x86-64, there are 4 levels of page tables (PGD → PUD → PMD → PTE).

⚡
Page Fault

If a process accesses a virtual address with no physical mapping yet, the CPU fires a page fault exception. The kernel's fault handler wakes up, either loads the page from swap/disk or kills the process (segfault!). This is how demand paging works — pages are loaded only when touched.

♻️
TLB — Translation Cache

Page table walks are expensive (4 memory lookups). The CPU caches recent translations in a TLB (Translation Lookaside Buffer). Context switches often need a TLB flush — a major reason context switching has overhead. Modern CPUs have per-process ASID tags to reduce flushes.

What is a segfault really?
A segmentation fault happens when a page fault can't be satisfied — either the virtual address has no mapping (null pointer dereference), or a process tries to write to a read-only page (writing to string literals), or accesses kernel space from userspace. The kernel sends SIGSEGV to the process, which kills it.
04 · virtual file system

VFS — "Everything is a File"

Linux's most famous idea: everything is a file. Your disk file? File. A network socket? File. The keyboard? File (/dev/input). A running process? File (/proc/1234). The VFS is the abstraction layer that makes this magic work.

🗂️
inode — the real identity

Every file has an inode — a data structure stored on disk with the file's metadata: permissions, owner, size, timestamps, and block pointers (where the actual data lives on disk). The filename is not stored in the inode — it's stored in the directory. That's why hard links can give one file multiple names.

📛
dentry — name to inode mapping

A dentry (directory entry) maps a filename to an inode number. When you do open("/home/user/main.c"), the kernel walks the path: root dentry → "home" → "user" → "main.c", resolving each to an inode. The dcache caches these so repeated opens are fast.

📂
file object — per-process handle

When a process opens a file, the kernel creates a file object in memory with the current position pointer (where in the file you are). The process gets back a small integer — the file descriptor (fd). fd 0=stdin, 1=stdout, 2=stderr. Your next open() gets fd 3.

🔌
VFS operations table

Each filesystem (ext4, BTRFS, tmpfs, procfs) registers a set of function pointers: read(), write(), seek(), etc. VFS calls these generically. Reading from /proc/cpuinfo calls a different function than reading from a real file — but your code sees the same read() call.

🧠
Why is /proc "magic"?
/proc is a pseudo-filesystem — it doesn't exist on disk. It's generated on-the-fly by the kernel. When you cat /proc/meminfo, the kernel generates that text dynamically. /proc/1234/maps shows the memory map of PID 1234. /proc/cpuinfo reads actual CPU model registers. It's a live window into the kernel's internal state.
05 · hardware interrupts

Interrupts — How Hardware Talks to the Kernel

Hardware doesn't wait to be polled. When your keyboard key lands, the keyboard controller doesn't wait for the kernel to ask "any keys pressed?" — it shouts. That shout is an interrupt. The CPU immediately drops whatever it's doing and handles it.

📋
What is an IRQ?

IRQ stands for Interrupt ReQuest. Each hardware device has an IRQ number (keyboard=1, timer=0, disk=14...). When hardware fires, the CPU reads the IRQ number and looks it up in the IDT to know which handler to call. View your system's IRQs with cat /proc/interrupts.

📖
IDT — Interrupt Descriptor Table

The IDT is a 256-entry table in kernel memory. Each entry is a pointer to a handler function for that interrupt number. Entry 0 = divide-by-zero handler. Entry 1 = keyboard. Entry 14 = page fault. Entry 128 = syscall! The CPU register IDTR holds the address of the IDT.

⬆️
Top Half (fast)

When an interrupt fires, the kernel's top half handler runs with interrupts disabled. It must be blazing fast: acknowledge the hardware (so it stops asserting the IRQ line), save any critical data, then schedule the slow work. Usually takes <1 microsecond.

⬇️
Bottom Half (slow)

The heavier work happens in the bottom half — with interrupts re-enabled. Linux uses softirqs, tasklets, and workqueues. Example: NIC top half just grabs the packet from hardware. Bottom half runs TCP/IP stack, socket buffers — that's the slow expensive part.

🔧
APIC — Modern Interrupt Controller
Old systems used an Intel 8259 PIC — single chip, 16 IRQ lines. Modern x86 uses APIC (Advanced Programmable Interrupt Controller). Each CPU core has a Local APIC for local timers and IPI (inter-processor interrupts). The I/O APIC routes device IRQs to specific CPU cores. On an 8-core machine, the kernel can spread disk IRQs to core 2, network to core 5, etc. for parallelism.

06 · system calls

Syscalls — The Gate Between Worlds

Every time your program needs something from the kernel — reading a file, allocating memory, creating a thread — it makes a system call. It's the one legal way for user code to enter kernel mode. Think of it as ringing the front desk bell.

🔔
How a syscall works

Your program puts the syscall number in register rax, arguments in rdi, rsi, rdx..., then executes the syscall instruction. The CPU switches to Ring 0, jumps to the kernel's syscall entry point, which reads rax and dispatches to the right handler function.

📊
The Syscall Table

The kernel has a table — sys_call_table — an array of function pointers indexed by syscall number. sys_call_table[0] = sys_read. sys_call_table[1] = sys_write. There are ~340 syscalls in Linux x86-64. View them in /usr/include/asm/unistd_64.h.

🔬
strace — spy on syscalls

strace ls shows every syscall ls makes. It's eye-opening — a simple ls calls execve, dozens of mmaps, openat, getdents64, write, and exit_group. strace intercepts syscalls using the ptrace mechanism (same thing gdb uses).

💰
Syscall cost

Syscalls aren't free. Each one is a Ring 3 → Ring 0 transition, saves registers, does work, restores state, and returns. About 100–300ns each. Spectre/Meltdown mitigations (KPTI) made it 5–10× more expensive on older CPUs. High-performance code batches I/O to minimize syscalls.

Putting it all together
When you run cat file.txt — here's the full journey: shell forks (clone syscall) → child calls execve("/bin/cat") → kernel loads the ELF binary → cat calls open() → VFS resolves the path → inode lookup → fd returned → read() loop → VFS → ext4 driver → disk I/O or page cache hit → data copied to buffer → write(1, ...) → terminal driver → characters on screen. Every step goes through the kernel.
